1. Introduction

This document provides a comprehensive overview of the Enprotec Workflow Management System. The application is designed to digitize and streamline key operational processes, including internal stock requests, inventory management, and the handling of salvaged parts. Its goal is to provide a single source of truth, improve accountability, and create a complete audit trail for all operational workflows.

2. User Guide

This section explains how to perform daily tasks within the system. It is organized by the primary roles within the company to help you find the information most relevant to your job.

2.1 Getting Started: Logging In

  • Access: Navigate to the application's URL. You will be presented with a login screen.
  • Credentials: Use the email address provided to you by the system administrator.
  • Default Password: The default password for all users is P. You can see the password as you type by clicking the "eye" icon.
  • Dashboard: Upon successful login, you will be taken directly to your personalized Dashboard. The pages you see in the sidebar on the left are determined by your assigned role.

2.2 Role-Specific Guides

For the Site Manager, Project Manager, or Requester

Your primary role is to request stock and confirm its delivery.

  1. Creating a Stock Request: From the Dashboard or another relevant page, click a "New Stock Request" button. A form will appear.
    • Fill in the destination site and date required.
    • Use the searchable dropdown to find and add one or more parts.
    • Optionally, attach a picture (e.g., a broken part) by clicking "Choose File". On mobile, this will let you use your camera.
    • Optionally, add comments for the approvers.
  2. Tracking Your Request: You can see the status of all your requests on the "Workflows" page.
  3. Accepting/Rejecting Delivery: Once an item has been physically delivered, it will appear on your personal "My Deliveries" page. This is your inbox for final confirmation.
    • Click "Accept Delivery" if everything is correct. The workflow is now complete.
    • Click "Reject" if there is a problem (e.g., damaged item, wrong quantity). You must provide a comment explaining the issue. The request will then be sent to the "Returns" page for the Stock Controller to process.

For the Stock Controller

You manage the flow of inventory, from approvals to returns.

  1. First-Level Approval (Requests Page): New stock requests appear on the "Stock Requests" page.
    • Review each request. The "Qty on Hand" is shown to help you make a decision. An insufficient stock warning will appear in yellow.
    • Click "Approve" to send it to the Equipment Manager for final approval.
    • Click "Decline" and provide a reason if the request is invalid.
    • Use the inline comments to communicate with the requester or manager.
  2. Picking (Picking Page): Once a request is fully approved, it appears on the "Picking" page. This is your to-do list for the warehouse. After physically preparing the items, click "Mark as Picked & Loaded".
  3. Handling Returns (Returns Page): Deliveries rejected by end-users appear on the "Returns" page.
    • Review the item and the reason for rejection.
    • Click "Book Back into Stock". A pre-filled Stock Intake form will appear. Verify the details and submit to add the item back to your inventory.
  4. Inventory Management (Stores Page): Use this page to view all inventory. The "Receive Stock" button opens a form to log new incoming deliveries.

For the Equipment Manager

You provide the final approval for stock requests and manage the salvage process.

  1. Final Approval (Manager Approvals Page): Requests approved by the Stock Controller appear on the "Manager Approvals" page.
    • Perform a final review of the request and stock levels.
    • Click "Approve for Picking" to release the request to the warehouse for fulfillment.
    • Click "Decline" and provide a reason if necessary.
  2. Salvage Decisions (Salvage Store Page): Items booked into salvage appear on this page, awaiting your decision.
    • For each item, you can choose to "Mark for Repair" or "Mark for Scrap".
    • After the physical repair or scrap has been done, you will return to this page to confirm it, which moves it to the final "Close Out" stage for the Stock Controller.

For the Driver & Security

Your work is focused on the "Deliveries" page.

  1. Dispatching Items: Items ready to leave the warehouse appear in the "Ready for Dispatch" section.
    • Click "Sign Gate Release & Dispatch". A form will appear to capture dispatch details. Submitting this form moves the item to the "In Transit" section.
  2. Confirming Delivery: Once the items have reached their destination, find the request in the "In Transit" section.
    • Click "Confirm Delivery (EPOD)". A form will appear to confirm receipt. Submitting this sends the request to the original requester's "My Deliveries" page for final acceptance.

For Admins & Operations Managers

You have oversight and management capabilities across the entire system.

  1. Full Visibility: You have access to nearly all pages and can monitor all ongoing workflows.
  2. User Management (Users Page): You can add new users, edit existing user details (name, email, role, site), and activate or deactivate user accounts.
  3. Site Management (Sites Page): You can add new destination sites and freeze or activate existing ones. Freezing a site removes it from dropdowns for new requests but preserves historical data.

2.3 User FAQ (Frequently Asked Questions)

Q: What happens if I reject a delivery?
A: The request is sent to the "Returns" page. A Stock Controller will review your rejection reason and book the item back into the system's inventory.

Q: How do I track the status of my request?
A: Go to the "Workflows" page. You can find your request and see a visual indicator of its current stage in the process.

Q: What does "Frozen" mean for a site?
A: A frozen site is temporarily disabled. It will not appear as an option for new stock requests, but all past requests and data associated with it are kept for reporting purposes.

Q: Why can't I see a certain page in the sidebar?
A: The pages you can see are determined by your user role. For example, only Stock Controllers can see the "Picking" page. If you believe you are missing access you need, please contact a system administrator.

3. Technical Support Guide

This section is intended for IT and technical support staff. It provides an overview of the system's architecture, database schema, and troubleshooting procedures.

3.1 System Architecture

The application is a modern single-page application (SPA) with a simple but robust architecture:

  • Frontend: Built with React 19 and TypeScript. It uses the ESM.sh CDN via an import map in index.html, meaning there is no complex local build step required for dependencies.
  • Backend & Database: Powered by Supabase.
    • Database: A PostgreSQL database hosts all application data.
    • Storage: Supabase Storage (in the "Enprotec" bucket) is used for handling file uploads.
    • API: The application communicates with the database via the Supabase client library (@supabase/supabase-js). The connection details are configured in supabase/client.ts.
  • AI Services: Report summaries are generated by the configured AI provider (gemini-2.5-flash model), configured in services/geminiService.ts.
  • Styling: **Tailwind CSS** is used for all styling via a CDN link in index.html.

3.2 Database Schema Deep Dive

All application data is stored in the public schema. The tables are prefixed with en_. Below are the key tables and their relationships.

Table Name Description Key Relationships
en_users Stores user credentials, roles, and profile info. One-to-many with en_workflow_requests (as requester), en_workflow_comments, en_stock_receipts.
en_sites The master list of all destination sites. One-to-many with en_workflow_requests and en_inventory.
en_stock_items The master list of unique parts (part number, description, etc.). One-to-many with en_inventory, en_workflow_items, en_stock_receipts, en_salvage_requests.
en_inventory Tracks the quantity of each stock item in each store and at each site. Belongs to one en_stock_items and optionally one en_sites.
en_workflow_requests The core table for all requests. Tracks status, requester, site, etc. Has many en_workflow_items and en_workflow_comments. Belongs to one en_users and one en_sites.
en_workflow_items The line items for each request. Belongs to one en_workflow_requests and one en_stock_items.
en_workflow_comments Stores all user comments linked to a specific workflow request. Belongs to one en_workflow_requests and one en_users.
en_stock_receipts An audit log of every time stock is received. Belongs to one en_stock_items and one en_users.
en_salvage_requests Tracks items moving through the salvage workflow. Belongs to one en_stock_items and one en_users.
Database Views: The application makes heavy use of Database Views (e.g., en_workflows_view, en_stock_view) to pre-join tables. This simplifies frontend queries and improves performance. If a field appears missing in the app, checking the definition of the relevant view in the database schema is a good first step.

3.3 Workflow & Business Logic

Main Internal Request Flow Status Transitions:

  1. Request Submitted (by Requester on Stock Request Form)
  2. Awaiting Equip. Manager (by Stock Controller on Requests page)
  3. Awaiting Picking (by Equipment Manager on Manager Approvals page)
  4. Picked & Loaded (by Stock Controller on Picking page)
  5. Dispatched (by Driver/Security on Deliveries page via Gate Release form)
  6. EPOD Confirmed (by Driver/Site Manager on Deliveries page via EPOD form)
  7. Completed (by original Requester on My Deliveries page)

Rejection Paths:

  • A request can be set to Request Declined from the Requests or Manager Approvals pages. A reason is required.
  • A delivery can be set to Rejected at Delivery from the My Deliveries page. This moves it to the Returns page for processing.

Automatic Stock Deduction Trigger:

A critical piece of server-side logic is the PostgreSQL trigger named on_dispatch_trigger. This function automatically fires when a request's status is updated to 'Dispatched'. It then deducts the requested item quantities from the inventory record in the 'Operations Store'. This ensures stock levels are updated accurately without relying on the client application.

3.4 Troubleshooting Guide

Problem: User cannot log in.
Solution: 1. Verify their email address is correct in the en_users table in Supabase. 2. Remind them that the default password for all users is P. The password in the database is currently stored in plain text.

Problem: A user cannot see a page they are supposed to.
Solution: Check the viewPermissions object in App.tsx. The user's role must be listed in the array for the specific view they are trying to access. The sidebar visibility is controlled by this same object.

Problem: Data is not updating after an action (e.g., an approved request still shows up).
Solution: This indicates an issue with the state-refresh mechanism. 1. Check the component where the action is performed (e.g., Requests.tsx). Ensure the onDataChange() prop is being called after a successful Supabase update. 2. Check the component that is displaying stale data (e.g., Picking.tsx). Ensure it is listening to the dataVersion prop in its useEffect hook to trigger a re-fetch.

Problem: A form submission is failing silently or with a generic error.
Solution: 1. Open the browser's Developer Tools (F12) and check the Console tab for any red error messages from the application code. 2. Check the Network tab. Look for the Supabase API call (it will start with ndjaufwwbtekysvrmhwm.supabase.co). Check its response for a more detailed database error message. 3. Check the database logs directly in the Supabase Dashboard for the most detailed error information.

Problem: The application is crashing on load or showing widespread TypeScript errors after a code change.
Solution: This almost always points to an issue with the auto-generated types from Supabase. 1. The most likely culprit is supabase/database.types.ts. It may be out of sync with the actual database schema. 2. The standard fix is to use the Supabase CLI to regenerate this file from your database schema: npx supabase gen types typescript --project-id YOUR_PROJECT_ID > supabase/database.types.ts.

Problem: A SQL script fails with ERROR: unsafe use of new value ... of enum type.
Solution: This is a known PostgreSQL transaction issue. Any script that adds a new value to an ENUM (like a new user role or workflow status) must be split into two parts. 1. Run the ALTER TYPE ... ADD VALUE command(s) first, in a separate transaction. 2. After it succeeds, run the rest of the script that uses the new value (e.g., INSERT statements or CREATE VIEW).
